# 顾问出差行程安排

为顾问规划从出发地到目的地的完整出差行程，输出可直接给出差人使用的 Markdown 行程单。核心目标是：满足拜访时间约束，遵守差旅政策，给出可执行交通方案、酒店建议、预算和检查清单。

## 快速流程

1. 补齐必要输入：出发地址、目的地址、出发日期；有返回行程时补返回日期；有拜访约束时补拜访时间和地址。
2. 读取 `references/travel_policy.md`，严格执行舱位、座席、酒店和补贴标准。
3. 读取 `references/user_preferences.md`，在不违反公司政策和拜访时间约束的前提下应用个人航司偏好、无锡出发和虹桥换乘经验；涉及无锡出发时，必须显式查询并比较 MU/FM 虹桥接驳方案。
4. 读取 `references/flight_source_issues.md`，检查航班数据源已知缺口，尤其是 FlyClaw 可能漏查上海虹桥 SHA 的 MU/FM 航班。
5. 能使用 MCP 时，按“地图 -> 航班/高铁 -> 酒店 POI -> 路线时间”的顺序查询；不能使用时，明确标注“待实时确认”并给出保守方案。
6. 读取 `references/routing_rules.md`，比较直飞、经停、联程、高铁、临近城市接驳方案。
7. 用 `scripts/itinerary_planner.py` 生成或校验时间线、预算和 Markdown。
8. 输出行程单、推荐理由、备选方案、费用预算、检查清单、需用户确认事项。

## 必要输入

缺失必要输入时先询问，不要凭空编造。

| 字段 | 必填 | 说明 |
|------|------|------|
| 出发地址 | 是 | 具体门牌或办公地点，不能只写城市 |
| 目的地址 | 是 | 客户/项目现场/酒店附近具体地址 |
| 出发日期 | 是 | YYYY-MM-DD 或用户明确日期 |
| 返回日期 | 否 | 无返回时按单程出差处理 |
| 出发时间偏好 | 否 | 上午/下午/晚上/不限 |
| 拜访安排 | 否 | 日期、时间、客户名、地址 |
| 出差人/目的 | 否 | 无则用“-”和“客户拜访/出差” |

## MCP 使用

优先使用这些 MCP 服务。服务不可用时继续完成行程，但必须在输出中标注哪些数据需要实时确认。

| MCP 服务 | 用途 | 最低可接受输出 |
|---------|------|---------------|
| `amap-maps` | 地理编码、驾车/公交路径、酒店 POI | 经纬度、距离、耗时、汉庭门店候选 |
| `flight-ticket-server` | FlyClaw 聚合航班查询，兼容原航班 MCP 名称；覆盖国内/国际航班、票价、时刻、中转 | 航班号、起降机场、起降时间、经济舱票价、经停/中转信息、数据来源；余票/库存无法确认时标注待确认 |
| `12306-mcp` | 高铁车次、二等座价格和余票 | 车次、起终站、时间、二等座票价/余票 |

### Codex 沙盒网络处理

在 Codex 环境中，如果航班查询诊断包含 `NameResolutionError`、`nodename nor servname provided`、`Could not resolve host`，且环境变量中存在 `CODEX_SANDBOX_NETWORK_DISABLED=1`，不得直接判定航班不存在，也不得立即改用网页兜底。

处理顺序：

1. 先说明这是 Codex 沙盒禁用 Python 外网 DNS 的执行环境问题。
2. 使用同一 FlyClaw 查询命令在沙盒外重试（请求 `require_escalated`），保留原始机场约束和日期。
3. 如果沙盒外 FlyClaw 返回 `fliggy_mcp`、FR24、Airplanes.live 等结果，必须以该 MCP/FlyClaw 结果作为主航班数据源。
4. 只有沙盒外 FlyClaw 仍然失败或返回数据明显不完整时，才使用携程、航司、飞常准、Airportia、Trip.com 等公开网页作为补充核验，并在输出中标注“网页兜底”。
5. 最终行程单的数据来源说明必须区分：沙盒内失败、沙盒外 MCP/FlyClaw 成功、网页补充核验。

查询顺序：

1. 地理编码出发地址、目的地址，识别城市、机场、高铁站。
2. 查询出发地到目的地直飞航班，保留经济舱。
3. 按用户偏好单独查询东方航空（MU）/上海航空（FM）候选；出发城市为无锡时，即使无锡本地已有直飞，也必须查询“无锡东 -> 上海虹桥站 -> 上海虹桥机场 T2 -> 目的地机场”的 MU/FM 方案，并计算换乘缓冲。
4. 查询同航班号经停航班，记录经停机场和停留时长。
5. 无合适直飞/经停时查联程航班，中转窗口参考 `references/routing_rules.md`。
6. 同时查高铁直达/换乘，用二等座方案作为备选。
7. 无合适方案时查临近城市航班 + 1 小时内高铁接驳。
8. 查询目的地附近汉庭酒店，优先离首个拜访地址最近。
9. 计算每段市内交通距离、耗时和打车估算。

## 偏好航司强制执行

- 东方航空（MU）/上海航空（FM）不是普通备注，必须形成独立候选池。
- 出发城市为无锡时，不得因为“无锡本地有直飞”而跳过上海虹桥 MU/FM 方案。
- FlyClaw 等聚合源若返回结果中没有 MU/FM 或缺少上海虹桥（SHA）航班，必须用至少一个补充航班源按“航班号”或“精确机场对”核验，不能直接认定没有偏好航司方案。
- 已知问题：FlyClaw 查询 `上海/SHA -> 长春/CGQ` 可能只返回浦东 PVG 结果，漏掉 `MU5593` 等上海虹桥 T2 东航航班；遇到上海多机场城市时必须补查 `references/flight_source_issues.md` 中记录的核验路径。
- 最终输出必须包含“偏好航司核验”说明：列出查到的 MU/FM 候选、接驳可行性、是否采用；未采用时必须写明原因（例如到达过晚、当天无法稳妥接驳、成本/风险显著更高、票价/库存待确认）。
- 若偏好航司方案与最短/最便宜方案冲突，应先判断拜访可达性和换乘风险，再给推荐；不能只按价格或直飞排序。

## 强制政策

详细政策见 `references/travel_policy.md`。核心约束：

- 飞机只能推荐经济舱。
- 高铁只能推荐二等座。
- 酒店只能推荐目的地/拜访地附近的汉庭大床房。
- 航班优先级：直飞 > 经停同票 > 联程 > 临近城市 + 高铁接驳。
- 经停停留或联程中转停留超过 2 小时的航班直接过滤，不作为推荐或备选。
- 红眼航班默认不推荐，除非用户明确接受。
- 个人偏好为软约束：东方航空/上海航空优先；无锡出发时可采用无锡东 -> 上海虹桥高铁接驳后转飞的方案。

## 输出要求

最终输出使用中文 Markdown，结构固定：

1. 基本信息
2. 推荐方案摘要（说明为什么推荐）
3. 交通方案对比（Top 3 或说明无数据）
4. 偏好航司核验（MU/FM 候选及取舍）
5. 详细时间线
6. 酒店建议
7. 费用预算表
8. 出差检查清单（使用 `references/checklist_templates.md`）
9. 风险与待确认事项

交通方案对比表必须包含“余票/库存”列：高铁显示二等座余票，飞机显示经济舱余票或库存状态。实时数据缺失时，仍要输出可执行草案，并把“票价/余票/酒店房态/航班当日变动”列入待确认事项。

## 脚本辅助

`scripts/itinerary_planner.py` 可用于计算时间线、预算并渲染 Markdown：

```bash
python3 skill/scripts/itinerary_planner.py --input trip.json --output itinerary.md
```

输入 JSON 字段可参考脚本中的 `demo_trip()`。当 MCP 已经查到真实数据时，把航班/高铁/酒店/市内交通结果写入 JSON 再运行脚本；当没有实时数据时，可用脚本生成带“待确认”标记的草案。

## 质量检查

交付前自检：

- 日期和星期是否正确。
- 出发到机场/车站是否包含市内交通耗时和缓冲。
- 到达后是否留出取行李/出站/打车时间。
- 拜访时间是否可达，不能可达时明确提示冲突。
- 预算是否区分“已查实”和“待确认”。
- 交通方案对比是否显示高铁二等座余票、飞机经济舱余票/库存；无法获取时是否标注待确认。
- 所有推荐是否符合经济舱、二等座、汉庭大床房政策。
- 是否建立了 MU/FM 偏好航司候选池；无锡出发时是否核验了上海虹桥 T2 方案。
- 是否应用了 `user_preferences.md` 中的航司偏好和虹桥换乘经验；未采用时是否说明原因。
