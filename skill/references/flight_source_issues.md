# 航班数据源问题与执行复盘

本文件记录航班 MCP/FlyClaw 在差旅行程规划中暴露的数据缺口和对应执行规范。后续新窗口测试或正式规划时，必须先阅读本文件，再生成行程。

## 2026-05-09 复盘：无锡到长春测试漏查 MU5593

### 测试场景

- 用户需求：下周三从无锡帆软软件有限公司出发，去长春吉林银行总行。
- 当前日期：2026-05-09。
- 出发日期解析：2026-05-13。
- 用户偏好：优先东方航空（MU）/上海航空（FM）；无锡出发时可接受无锡东 -> 上海虹桥站 -> 上海虹桥机场 T2 转飞。

### 发现的问题

FlyClaw 查询 `上海 -> 长春`、`SHA -> CGQ`、`PVG -> CGQ` 时，返回结果均来自 `fliggy_mcp`，且都显示为 `PVG -> CGQ`，没有返回上海虹桥 `SHA -> CGQ` 的东航航班。

实际通过飞常准类航班源核验，`MU5593` 存在：

| 字段 | 信息 |
|------|------|
| 航班号 | MU5593 |
| 航司 | 中国东方航空 |
| 日期 | 2026-05-13 |
| 出发 | 上海虹桥 SHA T2 |
| 到达 | 长春龙嘉 CGQ T2 |
| 计划时间 | 08:55 -> 11:35 |
| 状态 | 计划 |
| 准点率 | 96.67% |
| 餐食 | 有餐食 |

同一 `SHA -> CGQ` 机场对还核验到：

- MU5679：15:50 -> 18:30，上海虹桥 -> 长春龙嘉。
- MU5595：15:50 -> 18:30，上海虹桥 -> 长春龙嘉。

### 原因判断（初判）

这不是 MU5593 不存在，而是 FlyClaw 当前聚合链路在该航线上的覆盖缺口：

- 城市级查询 `上海 -> 长春` 只返回浦东 PVG 结果。
- 旧版 FlyClaw 本地包装在构造飞猪查询时会把 `SHA` 转成“上海”，导致精确机场查询被降级为城市级查询。
- FlyClaw 适合做票价聚合和普通航线初筛，但不能作为 MU/FM 偏好航司的唯一事实源。

## 2026-05-09 二次复盘：飞猪源本身支持 SHA 精确查询

### 直接调用飞猪源的实验结论

绕过 `flyclaw.py` 的城市名转换，直接调用 `FliggyMCPSource.query_by_route()` 后确认：

| origin 入参 | 返回情况 |
|------|------|
| `SHA` | 返回 `MU5679 SHA -> CGQ 15:50 -> 18:30 ¥790`、`MU5593 SHA -> CGQ 08:55 -> 11:35 ¥840` |
| `上海虹桥` | 返回同样的 `MU5679`、`MU5593` 虹桥航班 |
| `虹桥` | 返回空 |
| `上海` | 返回浦东 PVG 航班 |
| `PVG` | 返回浦东 PVG 航班 |

### 根因修正

飞猪源并非天然漏查 `SHA -> CGQ`。根因在本地 FlyClaw route search 任务构建：

- `airport_manager.resolve_all("SHA")` 能正确解析为 `["SHA"]`。
- 旧版 `flyclaw.py` 在 Fliggy 任务中调用 `_iata_to_city_cn(origins[0])`，把 `SHA` 转成“上海”。
- 飞猪收到“上海 -> 长春”后按城市级航线返回浦东 PVG 航班，导致虹桥 MU/FM 航班被主流程漏掉。

### 修复规范

- 精确机场查询必须保留 IATA 约束：`SHA -> CGQ` 应传给飞猪为 `SHA -> CGQ`，不得先转成“上海 -> 长春”。
- 只有用户输入城市级地点（如“上海”“长春”）时，才允许使用中文城市名调用飞猪城市级查询。
- `虹桥`、`浦东`、`上海虹桥` 等本地可解析为机场的输入，应先解析为 `SHA`/`PVG` 再调用飞猪。
- 对精确机场查询，返回结果必须校验 `origin_iata` / `destination_iata`；不匹配机场的航班不得进入主推荐，只能作为降级备选并说明原因。
- `sufficient_source: fliggy_mcp` 的提前返回，只能在飞猪结果满足机场约束且带价格时触发。

## 强制执行规范

### 无锡出发必须做偏好航司核验

只要出发城市是无锡，必须执行以下查询，不得因为无锡本地已有直飞就跳过：

1. 无锡本地机场航班：`WUX -> 目的地机场`。
2. 上海城市级航班：`上海 -> 目的地城市`。
3. 上海虹桥精确机场对：`SHA -> 目的地机场`。
4. 如用户或经验指出具体航班号，按航班号核验，例如 `MU5593`。
5. 无锡东 -> 上海虹桥高铁，核验二等座余票和到站时间。
6. 上海虹桥站 -> 虹桥机场 T2 换乘缓冲，按最少 60 分钟、托运行李 90 分钟判断。

### 数据源优先级

- FlyClaw：用于航班价格、常规航线、普通直飞/中转初筛。
- 飞常准/Variflight 类工具：用于航班号、机场对、航站楼、状态、准点率、机型、餐食和偏好航司漏查核验。
- 12306：用于无锡东 -> 上海虹桥等高铁接驳余票和票价。
- 高德地图：用于市内交通和酒店 POI。

### 输出必须包含的说明

最终行程单必须有“偏好航司核验”段落，至少包含：

- 查询到的 MU/FM 候选航班。
- 是否从上海虹桥 T2 出发。
- 无锡东到上海虹桥的高铁接驳是否满足 60/90 分钟缓冲。
- 是否采用偏好航司方案。
- 若未采用，明确原因，例如：当天赶早班风险高、需前一晚住上海、到达过晚、票价/库存无法确认、总成本显著更高。

### 禁止行为

- 禁止只因 FlyClaw 没有返回 MU/FM，就写“无 MU/FM 方案”。
- 禁止只看城市级 `上海 -> 目的地` 查询，不查 `SHA -> 目的地机场`。
- 禁止把“无锡本地直飞”作为跳过上海虹桥 MU/FM 核验的理由。
- 禁止推荐非偏好航司后不解释为什么未采用 MU/FM。
